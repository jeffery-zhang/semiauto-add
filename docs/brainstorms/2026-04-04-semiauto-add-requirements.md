# Semi-Auto Add

## Problem Frame
现有 [`auto-add`](/D:/Code/Projects/auto-add) 把 OpenAI 授权、验证码和写号流程揉进了 `Node.js + Playwright` 自动化，维护成本高，失败点也集中在浏览器阶段。新的 [`semiauto-add`](/D:/Code/Projects/semiauto-add) 目标是把浏览器自动化彻底拿掉，改成“人手动完成网页操作，系统负责后端接口和邮件取码”的半自动流程，同时尽量复用 `auto-add` 已验证过的 HTTP 能力。

在此基础上，工具还需要扩展一个 `批量测试` 页签，用来批量测试已有账号、识别已封禁账号，并支持单删与批量删除。

## Requirements

**Operator Workflow**
- R1. 系统必须提供一个单页半自动操作界面，控件顺序固定为：邮箱输入框、生成 URL 按钮、生成结果区（展示 URL 和重新生成按钮，仅在生成成功后出现）、回调 URL 输入框、添加按钮。
- R2. 操作员输入邮箱后点击“生成 URL”时，系统必须调用与 [`auto-add/src/api/auth-url.js`](/D:/Code/Projects/auto-add/src/api/auth-url.js) 同源的 `GEN_AUTH_URL` 后端能力生成授权地址，并在界面上展示可复制、可打开的结果。
- R3. 邮箱输入值必须继续保留在流程里，但系统不得把“邮箱参与 `GEN_AUTH_URL` 请求”当作事实。该邮箱用于本次半自动任务的上下文和最终写号数据，而不是用于现有 `GEN_AUTH_URL` 接口入参。
- R4. 生成 URL 成功后，操作员必须手动打开该 URL，在外部页面自行完成授权过程；系统不再负责打开页面、点击按钮、监听 localhost 回调或自动填写网页表单。
- R5. 系统不再提供任何站内取码能力；操作员需要在外部流程中自行获取并处理授权相关信息，页面只负责接收最终回调 URL。
- R6. 操作员把完整回调 URL 粘贴到输入框后点击“添加”，系统必须从回调 URL 中解析 `code`，完成 `exchange_code` 与 `add_account`，并向操作员明确展示成功或失败。
- R6a. 最终回调 URL 的权威形态就是以 `http://localhost:1455` 开头，且其中必然包含 `code` 参数；新项目必须继续复用 [`auto-add/src/shared/callback-url.js`](/D:/Code/Projects/auto-add/src/shared/callback-url.js) 同等语义的提取逻辑，而不是改成新的解析规则。
- R7. 如果当前邮箱没有对应的待完成授权上下文、回调 URL 缺少 `code`、邮件里没有可用验证码，系统必须给出明确错误，而不是静默失败。
- R7a. 点击“重新生成”后，系统必须立即作废上一次生成出来的授权上下文，并清空旧的 code、回调 URL 与状态提示，避免把新旧会话混用。
- R7b. 一旦生成 URL 成功，若操作员修改顶部邮箱，系统必须等同于“重新生成前的重置”：清空旧 URL、旧 code、旧回调 URL 和旧状态，并要求重新生成。
- R7c. “生成 URL”“添加” 这两个异步动作都必须有明确的进行中状态；执行期间对应按钮必须禁用，并展示可读的 loading 文案。
- R7d. “添加” 成功后，页面必须保留本次成功结果摘要，直到操作员手动重新生成 URL 开启下一轮。
- R7e. 页面不再承担验证码读取职责，也不提供任何轮询或邮箱读取能力。

**Backend Reuse**
- R8. 新项目后端必须优先复用 [`auto-add`](/D:/Code/Projects/auto-add) 中已经独立于 Playwright 的 HTTP 能力，包括管理员 token 校验与刷新、`GEN_AUTH_URL`、`EXCHANGE_CODE_URL`、`ADD_ACCOUNT_URL`。
- R9. 新项目运行链路中不得包含 Playwright、浏览器驱动、页面 locator、自动填表、自动点按钮、自动抓 localhost 回调等任何浏览器自动化能力。
- R10. 系统必须保留完成 “添加” 所需的授权上下文，至少包括最近一次生成成功的 `auth_url`、`session_id`、`state` 和当前邮箱，并能确保这些值属于同一次生成结果。
- R10a. 授权上下文默认保存在前端 session 中；页面必须提供明确的“清除当前 session”能力，供操作员主动丢弃当前会话状态。

**Architecture Direction**
- R11. 项目应采用单体 Next.js 方案，由同一个应用同时承载页面 UI 与服务端 API，避免额外拆分独立前端仓库和独立 Node API 服务。
- R12. 前端只负责收集输入、展示结果和触发动作；所有密钥、管理员 token、接口调用与敏感响应都必须留在服务端。
- R13. 页面必须为每个动作提供独立的 loading、success、error 状态，以及输入未满足前的 disabled/blocked 状态，避免操作员误以为请求已完成、可重复提交，或当前步骤已经具备前置条件。
- R14. 该工具必须被定义为内部操作台，只允许受信任的内部操作员使用；具体接入控制方式可在 planning 决定，但需求层面不得把它当作公开匿名页面。
- R15. 服务端返回给前端的数据必须限制在当前操作所需范围内；不得把管理员 token 或第三方接口原始敏感响应直接透传给前端。
- R16. 默认日志不得记录管理员凭据；非调试场景下也不应完整记录回调 URL 原文，只保留定位问题所需的最小信息。
- R17. 服务端日志与前端可见错误信息不得泄露管理员 token、完整回调 `code` 或第三方接口原始敏感响应。
- R18. 新项目环境变量命名与含义必须直接复用 [`auto-add`](/D:/Code/Projects/auto-add) 当前约定，避免重新设计一套配置名。

**Batch Testing**
- R19. 页面顶部必须提供 tab 切换，至少包含 `添加账号` 与 `批量测试` 两个页签。
- R20. `批量测试` 页签必须通过 `GET {BASE_ROUTER_HOST}/api/v1/admin/accounts?page=<n>&page_size=100` 拉取账号列表，并根据返回的总页数继续拉完剩余页面，直到拿到全部账号数据。
- R21. 批量测试流程至少需要保留每个账号的 `id` 与 `邮箱` 作为后续测试和删除的主键数据。
- R22. 系统必须提供异步队列式的批量测试能力，对列表中的账号逐个或并发受控地调用 `POST {BASE_ROUTER_HOST}/api/v1/admin/accounts/<id>/test`，固定请求体为 `{"model_id":"gpt-5.4","prompt":""}`。
- R23. 单个账号的测试结果必须等到对应 EventStream 全部接收完成后才能判定；如果完整流中同时出现 `401` 与 `deactived` 文案，则该账号必须被标记为 `已封禁`。
- R24. 全部批量测试完成后，页面必须把本轮已测试的全部账号都渲染到表格中，而不是只展示 `已封禁` 账号。
- R25. 表格至少展示 `id`、`账号`、`状态`、`操作` 这些列，并支持按状态筛选。
- R26. 表格必须支持批量选择与批量删除。
- R27. 单个删除与批量删除都必须调用 `DELETE {BASE_ROUTER_HOST}/api/v1/admin/accounts/<id>`，并在成功后把对应账号从当前表格与本轮结果中移除。
- R28. 单个删除与批量删除都必须要求明确确认后才能执行，避免内部工具误删。
- R29. `批量测试` 页签必须提供独立的进度与结果反馈，至少能区分：列表加载中、批量测试中、批量测试完成、删除中、删除成功/失败。
- R30. 首版批量测试仍只需要支持当前接口契约，不要求跨多个不同测试模型，也不要求分页大小可配置。
- R31. 单个账号测试超时时间固定为 `30s`；超时账号必须以明确状态呈现在表格中，而不是静默消失。
- R32. 如果一个账号首次测试被判定为 `已封禁`，但后续手动再次测试成功，页面必须把该账号状态更新为测试成功。
- R33. `批量测试` 页签必须支持单个账号手动重新测试。
- R34. 批量测试结果必须至少区分 `已封禁`、`测试成功`、`测试失败` 三类状态，并提供对应计数。
- R35. 由于前端需要继续做状态筛选与分页，后端在当前服务生命周期内不得在测试完成后主动清理本轮批量测试结果。
- R36. 页面必须提供显式“清除批量测试数据”能力，由操作员主动清理当前批量测试结果与相关前端状态。

## Success Criteria
- 操作员可以在不启动 Playwright 的前提下，完整走通“生成 URL -> 手动授权 -> 粘贴回调 URL -> 添加账号”。
- 新项目只复用 `auto-add` 的 HTTP 和解析能力，不再依赖任何浏览器自动化代码。
- 对于失败场景，页面能明确指出是生成 URL 失败、回调 URL 无效，还是添加失败。
- 连续重新生成 URL 时，不会发生旧 `session_id` / `state` 与新回调 URL 混用。
- 三个异步动作在执行过程中都有清晰的 loading 和禁用状态，操作员不会误触发重复请求。
- 操作员可以在 `批量测试` 页签中完成：拉全账号列表、跑测试、识别已封禁账号、查看结果表格、执行单删和批量删。
- 账号只有在完整测试流同时出现 `401` 与 `deactived` 文案时，才会被归类为 `已封禁`。
- 测试完成后，表格展示的是本轮所有已测账号，并且可以按状态筛选。
- 对单个账号手动重测后，表格状态会按最新结果更新。

## Scope Boundaries
- 不做任何 Playwright 迁移兼容层。
- 不做授权页自动打开、自动填写、自动提交、自动抓回调。
- 不做多步骤向导或复杂后台；首版就是单页操作台。
- 不做站内取码、邮箱选择器或任何邮件读取 UI。
- 不保证跨页面刷新保留操作上下文；如果会话丢失，允许要求操作员重新生成 URL。
- 不做超过单页 `100` 条之外的接口契约变更；只是按现有分页接口把所有页都拉完。
- 不做封禁原因分类、测试重试策略配置、导出报表或定时任务。
- 不要求首版就实现复杂服务端持久化；当前服务生命周期内保留结果即可。

## Key Decisions
- 单体 Next.js：这个项目交互简单、服务端逻辑薄、敏感配置必须后置，拆成前后端两个独立应用只会增加部署和状态同步成本。
- 邮箱字段保留，但不伪造接口事实：现有 `GEN_AUTH_URL` 不吃邮箱，因此邮箱只作为任务上下文和最终写号入参使用。
- 回调改为手动粘贴：既然目标是彻底丢掉浏览器自动化，就不再监听 localhost 请求，而是让操作员把最终回调 URL 粘进来。
- 页面只处理授权 URL 与最终回调 URL，不承担任何邮件读取或验证码展示能力。
- 内部工具而非公开页面：实现时可以选择最轻的接入控制方式，但不能默认裸奔暴露管理员能力。
- 批量测试的封禁判定以完整 EventStream 为准：必须同时命中 `401` 与 `deactived` 两个文案，不能只凭单个片段提前下结论。
- 删除操作一律加确认：这是低成本但高价值的防误删保护。
- 批量测试表格展示全部已测账号，不只展示封禁账号；封禁只是其中一种状态。
- 批量测试结果默认保留，清理动作由前端显式按钮触发，而不是测试完成后自动清空。

## Dependencies / Assumptions
- [`auto-add`](/D:/Code/Projects/auto-add) 中现有 HTTP 模块可被抽离或复制到新项目中复用。
- Base Router 管理接口在新项目启动时仍然可用。
- 当前工具主要服务单个内部操作员，而不是公开多租户系统。
